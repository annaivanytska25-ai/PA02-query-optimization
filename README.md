# PA02-query-optimization

# Таблиці
У оптимізації запитів задіяні три таблиці : opt_orders, opt_clients, opt_products. Дані в них згенеровані за інструкцією до асайменту.

# Запити
Обидва запити повертають топ-10 активних клієнтів за кількістю замовлень продуктів категорії Category2 після 2022-10-08. Дані виводяться в таблицю із колонками з повним обʼєднаним імʼям клієнта (імʼя та прізвище), та кількість замовлень на обрану категорію із фільтрацією за датою. Результати також відсортовані по спаданню, від клієнта із найбільшою кількістю замовлень до найменшого.

# Головні введені зміни для оптимізації
Неоптимізований запит зʼєднує всі три таблиці, потім проходиться по фільтрам статусу, категорії та даті. Під час обробки запиту обробляються мільйони рядків даних, щоб у процесі знайти результат та відкинути непотрібні дані. Виконується послідовний перебір - seq scan усіх таблиць, дуже повільний процес.

Оптимізований запит містить в собі три CTE - filtered_orders, active_clients, category2_products. Кожна з них фільтрує таблицю окремо. Перша CTE фільрує замовлення за датою, друга за статусом, третя за категорією. Завдяки введеним індексам, усі три CTE повертають малий набір даних. Таким чином, при фінальному джойні зʼєднуються не усі дані із таблиць, а попередньо відфільтровані - через index scan.

# Індекси
У процесі оптимізації я обрала 5 індексів, щоб зкоротити обсяг даних, які обробляються. 
1. idx_opt_orders_order_date
Був обраний, тому що при фільтрації за датою скануються мільйони рядків у відповідній таблиці. Без індексу будуть скануватися усі рядки через seq scan, в той час як індекс дозволяє одразу перейти до потрібних даних через index scan.
2. idx_opt_orders_client_id
Був обраний з тих самих причин. Айди клієнта використовується при джойні двох таблиць. Без індексу, під час пошуку потрібного айді буде скануватися уся таблиця із мільйоном даних.
3.  idx_opt_orders_product_id
Знову та ж таблиця, ті ж причини - прибрати повне сканування усієї таблиці. Айді продукту також використовується при джойні двох таблиць.
4. idx_opt_clients_status
Був обраний, тому що під час пошуку клієнтів із активним статусом, перебираються усі рядки в таблиці. Використовується у CTE під час пошуку статусу.
5. idx_opt_products_category
Пошук категорії продукта використовується під час фільтрації y CTE. Без індексу перебирає усі продукти для пошуку, з індексом одразу знаходить потрібні дані.

# Результати explain analyze
В результаті аналізу query plan за допомогою explain analyze, запит було успішно оптимізовано. Порівнювати буду через показник execution time. Також хочу додати, що я запускала explain analyze декілька разів, щоб відзначити максимальний показник для непоптимізованого та мінімальний для оптимізованого запитів. Query plan неоптимізованого запиту показав, що execution time є 162ms. Оптимізований запит, із застосуванням індексів, має execution time 104ms. Не можу сказати, що зміни великі, але вони помітні.
